# Anthra Interview Scheduler - Current Implementation Handoff

Use this as context for discussing next steps/security with another assistant or developer. This describes the current built feature, including changes added during QA beyond the original requirements.

## Project Context

- App: Anthra Systems website, Astro frontend.
- Feature area: Firebase-based interview scheduler.
- Main routes:
  - Admin login: `/admin/interviews/login`
  - Admin dashboard: `/admin/interviews`
  - New schedule: `/admin/interviews/new`
  - Manage schedule: `/admin/interviews/manage?scheduleId={id}`
  - Bookings: `/admin/interviews/bookings?scheduleId={id}`
  - History: `/admin/interviews/history`
  - Candidate booking: `/interview?scheduleId={id}` or shared links generated by the app.

## Main Files

- `src/interview-scheduler/firebase.ts` - Firebase web config/init from `PUBLIC_FIREBASE_*` env vars.
- `src/interview-scheduler/auth.ts` - Firebase Auth admin login/logout/session helpers.
- `src/interview-scheduler/services.ts` - Firestore CRUD, schedule/slot/candidate/booking logic.
- `src/interview-scheduler/types.ts` - Scheduler types/interfaces.
- `src/interview-scheduler/utils.ts` - formatting, links, validation helpers.
- `src/interview-scheduler/pages/*.ts` - browser logic for admin/candidate pages.
- `src/interview-scheduler/scheduler.css` - scheduler UI styles.
- `Interview-scheduler/FIREBASE_SETUP.md` - Firebase setup notes.

## Admin Features Implemented

- Firebase Email/Password admin login.
- Protected admin routes; unauthenticated users are redirected to login.
- Header hidden on scheduler/admin pages for cleaner internal UI.
- Dashboard lists schedules with counts and actions.
- Create schedule with title, default role/opening, interview date, common meeting link, description, status, and opening titles.
- Schedule expiry is derived from interview date instead of manual expiry input.
- Manage schedule page supports:
  - activate/draft/close/archive actions,
  - editable common meeting link,
  - opening title management,
  - allowed candidate email management,
  - per-candidate opening/position selection,
  - slot creation and status management,
  - slot/email comparison panel.
- History/archive view exists for old/archived schedules.
- Delete schedule hard-deletes the schedule and known subcollections: allowed candidates, slots, and bookings.

## Candidate/Email Invite Features

- Allowed candidates can be added one-by-one or in bulk.
- Emails are normalized to lowercase/trimmed.
- Each candidate can have a specific opening title/position, selected from the schedule's opening-title dropdown.
- Candidate-specific invite link includes `email` query param so email is prefilled/frozen on candidate page.
- Dashboard has editable invite message settings:
  - subject template,
  - body template,
  - placeholders: `{{position}}`, `{{scheduleLink}}`, `{{meetingLink}}`, `{{email}}`, `{{scheduleTitle}}`, `{{interviewDate}}`.
- Invite message supports formatted copy:
  - `**bold**`, `<strong>bold</strong>`, and `<b>bold</b>` are converted for clipboard HTML.
  - URLs become clickable when pasted.
- Candidate row actions:
  - `Copy Message` copies formatted Gmail-friendly content.
  - `Open Gmail + Copy` opens Gmail compose with recipient/subject and copies formatted body; user pastes body manually because Gmail URL body does not preserve HTML formatting.

## Slot Features Implemented

- Slots are created from the schedule interview date plus selected start time.
- Admin selects duration instead of end time.
- Duration supports 5-minute increments up to 120 minutes.
- Slot preview shows computed date/time range.
- Whole time input is clickable where browser allows it.
- Past slots cannot be created.
- Overlapping slots show a warning but are allowed if admin confirms.
- Slot statuses supported: available, booked, blocked, cancelled.
- Slot can have its own meeting link; if absent, candidate sees the schedule common meeting link.
- Changing a booked slot to available/blocked/cancelled triggers booking release/cancel logic.

## Candidate Booking Features Implemented

- Candidate page checks schedule exists, status is active, and not expired/closed/archived.
- Candidate must enter/confirm an allowlisted email.
- Email is frozen after verification; duplicate editable email field was removed.
- Full name is compulsory.
- Mobile number is compulsory and must be exactly 10 digits.
- Resume/profile/message fields are optional.
- Available slots exclude booked, blocked, cancelled, expired, or already-past slots.
- Candidate can book only one slot per schedule.
- Booking uses Firestore transaction-style logic to prevent two candidates booking the same slot.
- If candidate already booked, page shows existing booking details instead of allowing another booking.
- Meeting link is shown after successful booking and when revisiting an existing booking; includes open/copy actions.
- Candidate-facing schedule expiry is hidden; candidate sees their booked slot date/time in readable Indian date format.
- Candidate errors use modal/popup style where practical instead of plain inline text.

## Admin Booking Workflow

- Admin bookings table shows slot, candidate name, email, phone, position, links, booking time, status, rating, selected flag, and notes.
- Admin can update:
  - candidate status: booked, interview done, selected, rejected, on hold, no show,
  - overall rating,
  - selected/not selected,
  - admin notes.
- Admin can cancel a booking and release the slot.
- When admin releases/cancels a booking, candidate `hasBooked` and `bookingId` are cleared so the candidate can book again.

## Edge Cases Covered in App Logic

- Missing Firebase config shows developer-friendly error.
- Invalid admin login shows error.
- Admin route access without login is blocked by frontend guard.
- Non-allowlisted email cannot proceed.
- Same email cannot book twice.
- Booked slots disappear from available slot list.
- Candidate cannot book a past slot.
- Candidate cannot book when schedule is draft/closed/archived/expired.
- Slot already booked during submit shows an error.
- Invalid 10-digit mobile number blocks booking.
- Removing booked candidate email is blocked unless booking is cancelled/released first.
- Deleting schedule also deletes related slots, allowed candidates, and bookings.
- Gmail direct body formatting issue was handled by using clipboard HTML + manual paste.

## Important Security Context

Current candidate booking is frontend + Firestore transaction based. It works functionally, but production security should not rely only on frontend checks.

Main concern:

- Candidates/public users interact directly with Firestore from the browser.
- Firestore rules must prevent public users from reading other candidates/bookings or writing admin-only fields.
- Without Cloud Functions, rules must allow some public reads/writes for booking, which is a compromise.

Best production option:

- Move booking into a callable Firebase Cloud Function, e.g. `bookInterviewSlot`.
- Function should check allowlist, schedule status/expiry, one booking per email, slot availability, and write booking/slot/candidate updates atomically.
- Then Firestore rules can be strict: admin reads/writes admin data; candidates cannot directly write bookings/slots/candidate flags.

No-Cloud-Function compromise:

- Tighten Firestore rules so admin can read/write all scheduler data.
- Public users can read only active schedule and available slot data required for booking.
- Public users can create booking documents only with strict field validation.
- Public users cannot read all bookings or write rating/status/admin notes.
- Candidate "already booked details" may need frontend adjustment if booking reads are locked down.
